## 3.if vs if-else
### 3-1.if-else
- 위에서부터 차례대로 검사, 처음 참이 되는 조건문만 실행하고 아래는 검사x
- 여러 조건 중 하나만 해당되는 경우 
- 학점 구하기, 메뉴 선택, 숫자가 양수 or 음수 or 0인지 판별
- ```c
  	int a;
  // if 여러 개를 써도 동작하지만 조건을 모두 검사하는 불필요한 연산수행
	printf("정수 입력: ");
	scanf_s("%d", &a);

	if (a > 0)
		printf("양수");
	else if (a < 0)
		printf("음수");
	else
		printf("0");
  ```
### 3-2.if
- 위의 조건이 참이어도 아래 조건도 계속 검사
- 여러 조건이 모두 의미 있을 때 적합
- 최댓값 갱신, 여러 조건 체크, 옵션 여러 개 동시 적용
## 4.컴파일
- C 컴파일은 전처리, 컴파일, 어셈블, 링크 단계로 이루어지며, GCC/Clang/MSVC 같은 컴파일러를 사용해 실행 파일 또는 임베디드용 바이너리를 생성
```
gcc main.c -o main // 실행 파일
gcc -E main.c -o main.i // 전처리만
gcc -S main.c -o main.s // 컴파일만 (C -> Assembly)
gcc -c main.c -o main.o // 어셈블만
gcc main.o -o main      // 링크 (오브젝트 파일을 실행 파일로 연결)

// 각각 컴파일
gcc -c main.c -o main.o
gcc -c led.c -o led.o
gcc -c uart.c -o uart.o
gcc main.o led.o uart.o -o app // 링크
```
## 4-1.컴파일러 종류
### GCC
- Linux에서 많이 사용
- Windows에서도 MinGW, MSYS2로 사용 가능
- 임베디드용 크로스 컴파일러도 많음
### MSVC
- 마소의 C/C++ 컴파일러
- Visual Studio에서 주로 사용
- ```
  cl main.c
  ```
### 임베디드용 크로스 컴파일러
- 크로스 컴파일: PC에서 AVR, STM32용 바이너리 생성 
- AVR: avr-gcc
- ARM Cortex-M: arm-none-eabi-gcc
- ```
  arm-none-eabi-gcc main.c -o main.elf // STM32 
  ```
### 디버그 빌드/ 릴리즈 빌드
- ```
  gcc -g main.c -o main // 디버그 빌드 (디버그 정보 포함, GDB 같은 디버거 사용 가능)
  ```
### 결과 파일 종류
- .i: 전처리 결과 파일
- .s: 어셈블리 파일
- .o: 오브젝트 파일
- .elf: 임베디드용 실행/디버그용 파일
- .hex: MCU 플래시에 올리기 좋은 텍스트 형식 바이너리 (데이터, 메모리 주소, 체크섬)
- .bin: 순수 바이너리 파일
- .exe: Windows 실행 파일
### PC 개발과 임베디드 개발의 차이
```
./main // PC 실행파일
```
```
arm-none-eabi-gcc main.c -o main.elf // 임베디드 C개발
```
- .elf 생성
- 필요하면 .bin, .hex 변환
- 보드에 다운로드
- 디버거(ST-LINK, J-LINK 등)로 실행
-임베디드는 컴파일 후 바로 PC에서 실행하는 게 아니라 보드에 올림

## 5.Makefile

## 6.CMAKE
- 빌드 규칙 파일이나 IDE 프로젝트를 만들어 주는 상위 도구
- CMakeLists.txt를 읽어서 Unix에서는 Makefile, Ninja 빌드 파일 생성, Visual Studio 솔루션 생성
- 직접 컴파일러가 아니고, 직접 Makefile 그 자체
- find_package()로 외부 패키지를 찾을 수 있
- ```
  cmake -S . -B build // 빌드 시스템 파일을 생성
  cmake --build build // 생성된 빌드 시스템을 이용해 실제 빌드 수행
  ```
## 6.CLI (Command Line Interface)
- 글자로 명령어를 입력해서 프로그램이나 시스템을 조작하는 방식
- GUI는 마우스로 폴더, 버튼 등 선택 
- ```
  cd project
  gcc main.c -o main
  ./main
  ```
### 6-1.임베디드에서의 CLI
1. PC의 명령어 창 (터미널에서 빌드, flash 다운로드, git 사용)
2. 보드 안의 CLI 기능은 UART로 연결하여 사용 (led on, help, status)
3. 장치 내부에 만든 명령어 인터페이스, 즉 MCU 프로그램 안에 간단한 셸을 만든

## 10. scanf vs fgets
- scanf: 공백 전까지만 입력 받음, 엔터\n는 저장x, 단어 1개, 버퍼 오버플로우 위험 존재  
- fgets: 줄 전체 입력 받음, 엔터 저장될 수 있음, 배열 크기만큼 안전하게 입력 받음, 문장용
## 오버헤드 vs 오버플로우
### 오버헤드
- 작업을 수행하기위해 추가로 들어가는 시간, 메모리, 처리량
- 성능 저하의 원인, 추가 부담
- ```c
  for (int i = 0; i < 1000; i++) {
    strlen(str); // 같은 값을 계속 계산 -> 쓸데없는 비용 증가 
  }
  ```
### 오버플로우
- 담을 수 있는 범위를 초과하는 것 (변수, 버퍼, 배)
- ```c
    char a = 127; // ~128 ~ 127, +1: 오버플로우 
    a = a + 1;
    ```
## 11.핵사덤프
- 메모리에 들어있는 데이터를 16진수(hex)로 보여주는 출력 방식
- 문자열, 구조체, 배열, 파일 데이터, 어떤 변수의 실제 메모리 값을 바이트 단위로 쪼개서 사람이 볼 수 있게 출력
- Hex = hexadecimal = 16진수
- dump = 쏟아내다, 그대로 보여주다
### 예시 1) 'A' 의 아스키 값
- 10진수: 65
- 16진수: 0x41
- hexdump: 0000  41                                               A
             ㄴ시작 위치, 실제 메모리값, A: 사람이 읽기 쉽게 문자로도 같이 보여줌
### 예시 2) "Hello" 
- 48 65 6c 6c 6f 00
- hexdump: 0000  48 65 6c 6c 6f 00                                Hello.
